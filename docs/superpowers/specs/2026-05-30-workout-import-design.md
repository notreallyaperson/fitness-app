# Workout Import ‚Äî Prefilled Session from JSON

**Date:** 2026-05-30
**Status:** Approved

## Goal

Let the user paste a JSON workout (generated by an external AI like ChatGPT) and
have the app create a **new session with every exercise and set pre-filled**, so
they can run the workout and just adjust actual reps/weight/effort as they go.

## JSON schema (what the user hands their AI)

```json
{
  "name": "Push Day A",
  "notes": "optional note for the session",
  "exercises": [
    {
      "name": "Bench Press",
      "metric_kind": "weight_reps",
      "sets": [
        { "weight": 60, "weight_unit": "kg", "reps": 8, "rpe": 8 },
        { "weight": 60, "weight_unit": "kg", "reps": 8, "rpe": 9 },
        { "weight": 62.5, "weight_unit": "kg", "reps": 6 }
      ]
    },
    {
      "name": "Plank",
      "metric_kind": "time_only",
      "sets": [{ "time_seconds": 60 }, { "time_seconds": 60 }]
    }
  ]
}
```

- `name` ‚Äî required, 1‚Äì80 chars. `notes` ‚Äî optional, ‚â§2000 chars.
- `exercises` ‚Äî required, ‚â•1 entry.
  - `name` ‚Äî required.
  - `metric_kind` ‚Äî optional. Used **only when the exercise must be auto-created**.
    If the name matches an existing catalog exercise, the catalog's metric_kind
    wins. If omitted and creation is needed, infer from set fields
    (`time_seconds` ‚Üí `time_only`, `distance` ‚Üí `distance_only`, else
    `weight_reps`).
  - `notes` ‚Äî optional, per-exercise.
  - `sets` ‚Äî array (may be empty). Each set mirrors the existing set fields and
    is validated by the existing `SetInputSchema`: `reps`, `weight`,
    `weight_unit` (`kg`|`lbs`), `time_seconds`, `distance`, `distance_unit`
    (`m`|`km`|`mi`), `rpe` (1‚Äì10), `is_warmup`, `notes`.
- `metric_kind` enum: `weight_reps`, `bodyweight_reps`,
  `weighted_bodyweight_reps`, `time_only`, `time_weight`, `distance_only`,
  `distance_time`, `none`.

Decision: explicit per-set array (not a `"sets": 3, "reps": 8` shorthand) so top
sets, backoffs, and warmups import precisely.

## Components

- **`WorkoutImportSchema`** (Zod, in `src/lib/validation.ts`) ‚Äî reuses
  `SetInputSchema` for each set.
- **Pure transform** (`src/lib/import-plan.ts`):
  - `inferMetricKind(sets)` ‚Äî metric_kind from set fields.
  - `buildImportPlan(parsed)` ‚Äî maps validated JSON to an insert-ready plan
    (exercises with resolved metric_kind, sets with explicit positions). No DB.
- **`importWorkout(plan)`** (`src/lib/db/import-workout.ts`) ‚Äî resolves each
  exercise by case-insensitive name match against the catalog; auto-creates a
  custom exercise when no match. Creates the session, batch-inserts
  `session_exercises` (ordered), then `sets` per row. On any insert failure
  after the session is created, deletes the session (no orphan) and rethrows.
- **`importWorkoutAction(jsonText)`** (`src/server/actions/sessions.ts` or a new
  `import.ts`) ‚Äî parse JSON ‚Üí validate ‚Üí `importWorkout` ‚Üí `redirect`
  to `/sessions/:id`. Throws readable messages for bad JSON / validation errors.
- **`/sessions/import` page** ‚Äî textarea + Import button, a collapsible sample
  and a copy-paste ChatGPT prompt so the user can generate conforming JSON.
  Errors surface via toast.
- **Entry point** ‚Äî "Import from JSON" link on `/sessions/start`.

## Error handling

- Unparseable JSON ‚Üí "Couldn't parse JSON".
- Zod validation errors ‚Üí readable first error (e.g. empty `exercises`,
  invalid enum).
- Unknown exercise names ‚Üí auto-created, not an error.

## Testing

- Unit: `WorkoutImportSchema` accepts valid input and rejects malformed
  (empty exercises, bad enum, bad rpe).
- Unit: `inferMetricKind` and `buildImportPlan` (positions, metric inference,
  set mapping, catalog metric override path is handled in DB layer).
- DB orchestration kept thin; not unit-tested here.

## Out of scope

- Shorthand set notation.
- Importing into a template (only sessions).
- File upload (textarea paste only).
